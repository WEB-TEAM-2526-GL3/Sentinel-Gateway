Partie I

Vue d'ensemble du produit

Ce que fait Sentinel Gateway aujourd'hui, ce qui est volontairement hors périmètre, et qui s'en sert.
Slide 01

1.1 — Positionnement et objectifs

Sentinel Gateway est un control plane NestJS qui pilote une instance Kong existante, observe son trafic via Prometheus et orchestre la réponse aux anomalies.

Kong — Data plane

Kong 3.9.1 en mode base PostgreSQL, plugin Prometheus activé, configuration de bootstrap déclarative (kong.yaml). Aucun code métier n'est embarqué dans Kong.

sentinel-core — Control plane

Application NestJS unique : configure Kong via son Admin API, lit les métriques exposées à Prometheus, applique des règles d'anomalie, ouvre des incidents collaboratifs et notifie les canaux externes (Slack, Discord).

Objectifs concrets du produit

ObjectifCe que Sentinel apporteModule porteur
Configurer KongAPI REST + GraphQL pour CRUD des services, routes, plugins et consumers, sans utiliser l'Admin API à la main.GatewayModule
MesurerPolling périodique des requêtes / latences / statuts Kong exposés à Prometheus, cache local par scope.MetricsModule
DétecterRègles d'anomalie configurables (error-rate, latence p95, santé upstream) avec cooldown.MonitoringModule
CoordonnerIncidents persistés, salle WebSocket avec présence, journal d'actions, SSE de notification.IncidentsModule
NotifierWebhooks sortants typés Slack / Discord / Generic, signature HMAC, deliveries journalisées.WebhooksModule
Slide 02

1.2 — Périmètre actuel

Dans le périmètre

  • CRUD complet des services, routes, plugins et consumers Kong via l'Admin API.
  • Polling Prometheus toutes les 15 s et cache par scope (consumer × service).
  • Règles de monitoring CRUD avec types ERROR_RATE, LATENCY_P95, UPSTREAM_HEALTH.
  • Création automatique d'incidents sur monitoring.threshold.exceeded.
  • Salle d'incident WebSocket (présence, chat, ack, resolve) et SSE incident.created.
  • Webhooks sortants Slack / Discord / Generic, signature HMAC, retries avec backoff.
  • Callback Meta Messenger entrant (vérification + ingestion en mémoire).
  • API GraphQL Apollo couvrant gateway, incidents, monitoring, métriques, webhooks.
  • Authentification JWT et secret CEO pour les actions sensibles.

Hors périmètre

  • Gestion de providers LLM (OpenAI, Gemini, etc.) et calcul de coûts par tokens.
  • Détection de prompts répétés et estimation de gaspillage.
  • Moteur de fallback automatique entre providers.
  • Budgets mensuels par service ou par provider.
  • Chiffrement AES-256-GCM des secrets de provider.
  • Streaming SSE des métriques (seul incident.created est diffusé en SSE).
  • Auto-résolution d'incident sans erreur pendant 5 minutes.
  • Redis Pub/Sub pour scaling multi-instances.
  • Persistance des webhooks et messenger events (V1 en mémoire).
Slide 03

1.3 — Acteur unique : l'Admin

Sentinel ne reconnaît qu'un seul rôle : l'Admin. Toute personne authentifiée a les mêmes droits, à l'exception des actions de désactivation d'un autre admin qui exigent en plus le CEO secret.

ActeurDescriptionInteractions principales
AdminIngénieur plateforme / SRE chargé de la supervision Kong.Configure Kong, gère les règles de monitoring, rejoint les salles d'incident, configure les webhooks.
Kong (système)Data plane proxifiant le trafic.Expose ses métriques à Prometheus via le plugin prometheus.
Prometheus (système)Source de métriques.Scrape Kong toutes les 15 s ; interrogé par MetricsService.
ExternesDestinataires des notifications.Slack, Discord et tout webhook GENERIC.
Meta MessengerÉmetteur de callbacks entrants.Vérification du token + livraison d'événements à /messenger/webhook.

Conséquence architecturale. AuthModule implémente une auth JWT simple ; les actions sensibles (suppression d'un admin) exigent en plus CeoSecretService.validateOrThrow. Pas de matrice RBAC à ce stade.

SSE — GET /incidents/sse

Flux incident-created ; permet à un dashboard non-WS de réagir à la création d'un incident sans rejoindre la salle.

2.4 — Modules NestJS et responsabilités

Les modules sont indépendants et communiquent par @nestjs/event-emitter. Chacun publie ses providers via son module NestJS ; la documentation détaillée vit dans sentinel-core/docs/.

ModuleResponsabilitéPersistance
GatewayModuleWrapper de l'Admin API Kong (services, routes, plugins, consumers) via SDK généré kong-client.Kong (externe)
MetricsModulePolling Prometheus toutes les 15 s, cache par scope, détection naïve de santé (≥ 10 fenêtres consécutives avec 5xx / 429 → service dégradé). Émet metrics.updated, metrics.poll.failed, health.changed.Mémoire
MonitoringModuleCRUD de règles (ERROR_RATE, LATENCY_P95, UPSTREAM_HEALTH), checks planifiés et manuels, cooldown par règle. Émet monitoring.threshold.exceeded.Postgres
IncidentsModuleCréation d'incident sur monitoring.threshold.exceeded, salle WebSocket /incident-room, journal d'actions, snapshot SSE. Émet incident.created.Postgres
WebhooksModuleFaçade sortante Generic / Discord / Slack via formatters dédiés, signature HMAC optionnelle, retries avec backoff, deliveries journalisées. Listener incident-created.Mémoire
MessengerModuleCallback Meta entrant : vérification du token, stockage des événements et PSIDs.Mémoire
SentinelGraphqlModuleSchéma Apollo unique exposant gateway, monitoring, incidents, métriques, webhooks et messenger.—
AuthModuleJWT (Passport), guard, CeoSecretService pour les actions de désactivation.Postgres
UsersModuleRéférentiel des admins (email, mot de passe haché, rôle, statut).Postgres
CommonModuleServices transverses : GenericService, notifications, sanitisation.—

Chaîne nominale. Prometheus → MetricsModule → MonitoringModule → IncidentsModule → WebhooksModule. Aucun module n'appelle directement le service d'un autre : ils s'écoutent via EventEmitter2.

WebSocketAdmin ↔ SentinelNamespace /incident-room (Socket.io) : présence, chat, ack, resolve.
WebhookSentinel → ExternePOST signé HMAC vers Slack, Discord ou endpoint GENERIC ; retries avec backoff.
PromQLSentinel → PrometheusPrometheusService assemble les PromQL des compteurs Kong et latences.
Slide 16

2.11 — Sécurité et authentification

Authentification

  • JWT via Passport (JwtStrategy, JwtAuthGuard).
  • GraphQL protégé par GqlJwtAuthGuard.
  • WebhookAdminGuard pour les routes d'administration des webhooks (clé X-Sentinel-Admin-Key).
  • CeoSecretService requis pour désactiver un autre admin.

Intégrité & sécurité opérationnelle

  • Signature HMAC SHA-256 des payloads webhook sortants si secret configuré.
  • Sanitisation centralisée dans common/sanitize.ts.
  • Validation stricte des DTO via class-validator.
  • Les secrets webhook ne sont jamais renvoyés dans les réponses publiques (hasSecret uniquement).
  • Le token Messenger ne quitte jamais l'environnement.

Points d'attention V1. TypeORM synchronize: true est activé : OK en dev, à désactiver en production. Les webhooks sont en mémoire : redémarrer le backend efface la configuration. Pas encore d'HTTPS au niveau Sentinel — Kong est censé gérer la terminaison TLS.

3.3 — Limites actuelles

En mémoire (perdu au redémarrage)

  • Webhooks et leurs deliveries.
  • Événements et recipients Messenger.
  • Cache de métriques et état de santé service.
  • Présence dans les salles d'incident.

Non implémenté en V1

  • Auto-résolution d'incident.
  • Provider Messenger sortant (uniquement inbound).
  • Migrations TypeORM (mode synchronize: true).
  • Redis Pub/Sub multi-instances.
  • RBAC fin (un seul rôle Admin + secret CEO).
  • Stream SSE des métriques.

Pistes d'évolution déjà câblées

  • Le bus d'événements EventEmitter2 peut être étendu à Redis pour multi-instances sans réécrire les listeners.
  • L'enum WebhookProvider peut accueillir MESSENGER sortant ; la spec de payload est déjà documentée (messenger_integration.md).
  • L'état DeliveryStatus.PENDING est réservé pour une future implémentation asynchrone par queue.
