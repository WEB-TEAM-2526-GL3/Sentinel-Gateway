Partie I

Cahier des charges détaillé

1.1 — Contexte et positionnement

Les organisations exposent un nombre croissant d'APIs internes et de fournisseurs externes — notamment des fournisseurs de modèles de langage (OpenAI, Gemini, Claude) et des services tiers (Stripe, Twilio). Ces dépendances introduisent trois problèmes opérationnels majeurs : l'absence de visibilité unifiée sur le trafic et les coûts, la difficulté à réagir vite lors d'une panne fournisseur, et l'impossibilité de gouverner les dépenses liées à l'IA.

Sentinel Gateway répond à ces besoins en se positionnant comme un control plane (plan de contrôle) intelligent. Le projet ne réinvente pas l'infrastructure de routage : il s'appuie sur Kong API Gateway, une solution éprouvée en production, qui conserve son rôle de data plane.

Kong — Data Plane

Routage HTTP, proxy, load balancing, gestion des upstreams, application des plugins (authentification, rate limiting), transformations de requêtes. Couche performante, stable, non modifiée.

Sentinel — Control Plane

Observabilité, monitoring temps réel, calcul des coûts, détection et gestion collaborative des incidents, gouvernance des dépenses IA, orchestration dynamique de la configuration Kong. Couche d'intelligence.

Objectif pédagogique et technique

Au-delà de sa valeur métier, le projet illustre l'usage pragmatique de cinq protocoles web, chacun retenu pour un besoin réel et distinct, sans sur-ingénierie : REST configuration CRUD   GraphQL exploration analytique   SSE flux de métriques   WebSocket collaboration incident   Webhook intégrations externes.

Objectifs du produit

1.2 — Périmètre du projet

Dans le périmètre (MVP)

  • Configuration des providers, services, budgets, règles de fallback et webhooks.
  • Synchronisation de la configuration vers Kong via son Admin API.
  • Ingestion des journaux Kong, calcul de latence / erreurs / coûts.
  • Détection des prompts répétés et estimation du gaspillage.
  • Diffusion temps réel des métriques (SSE) et collaboration incident (WebSocket).
  • API analytique GraphQL pour l'exploration historique.
  • Notifications externes par webhook (Slack, Discord).
  • Tableau de bord React : monitoring, configuration, salle d'incident.
  • Chiffrement des secrets fournisseurs (AES-256-GCM).

Hors périmètre (MVP)

  • Multi-tenant : une seule organisation logique.
  • Gestion fine des rôles : un seul type d'acteur (Admin).
  • Facturation client / refacturation par projet.
  • Cache sémantique des réponses LLM.
  • Résolution automatique d'incident assistée par IA.
  • Intégration service mesh (Istio).
  • Déploiement multi-région.
  • Prévision de coûts par apprentissage automatique.

Les éléments hors périmètre constituent la trajectoire d'évolution post-MVP. L'architecture (Partie II) est conçue pour les accueillir sans refonte.

1.3 — Acteur unique : l'Admin

Pour simplifier la logique métier du MVP, le système ne reconnaît qu'un seul type d'acteur : l'« Admin ». Il n'existe ni utilisateur final, ni client, ni séparation administrateur / utilisateur. Toute personne authentifiée sur Sentinel possède les mêmes droits.

ActeurDescriptionInteractions principales
AdminOpérateur de la plateforme : ingénieur plateforme, SRE ou développeur en charge de la supervision des APIs.Configure Kong, supervise le trafic, déclenche les fallbacks, collabore en salle d'incident, consulte les analyses.
Kong (système)Acteur système : émet les journaux de trafic consommés par Sentinel.Pousse les logs HTTP vers le Metrics Engine.
Systèmes externesActeurs système : destinataires des notifications (Slack, Discord, PagerDuty).Reçoivent les webhooks émis par Sentinel.

Conséquence architecturale. Le module AuthModule implémente une authentification JWT simple (connexion, émission, vérification du jeton) sans matrice de permissions. L'extension vers un modèle RBAC est documentée comme évolution.

1.6 — Cas d'utilisation détaillés

Trois cas d'usage centraux sont décrits sous forme de scénarios. Les autres (CRUD providers, services, webhooks) suivent le même patron CRUD standard.

UC-01 — Configurer un service avec fallback

ActeurAdmin
PréconditionLes providers OpenAI et Gemini existent et sont actifs.
Scénario nominal1. L'admin crée le service chat-service avec un budget mensuel. 2. Il associe OpenAI comme provider primaire. 3. Il associe Gemini comme provider de repli. 4. Il crée une règle de fallback on_error, on_timeout. 5. Sentinel propage la configuration vers Kong. 6. Le système confirme et journalise.
Scénario alternatif3a. Un second provider primaire est refusé (conflit 409, voir BR-02).
PostconditionLe service est routable via Kong ; la règle de fallback est armée.

UC-02 — Détecter un incident et basculer vers un provider de repli

ActeurAdmin (+ Kong et Metrics Engine comme acteurs système)
PréconditionUC-01 réalisé ; le trafic transite par Kong.
DéclencheurOpenAI dépasse le seuil de timeout (> 5 s) de façon répétée.
Scénario nominal1. Kong pousse le log d'erreur à Sentinel. 2. Le Metrics Engine détecte le timeout et émet ErrorDetected. 3. L'Incident Engine crée un incident et émet IncidentCreated. 4. Une salle WebSocket s'ouvre, les admins sont notifiés. 5. Le SSE diffuse l'événement au tableau de bord. 6. Un admin clique « Switch to Gemini » dans la salle. 7. Le KongAdapter modifie l'upstream. 8. L'événement FallbackActivated est diffusé et journalisé.
Scénario alternatif3a. Une règle de fallback automatique existe : l'étape 6 est exécutée sans intervention humaine.
PostconditionLe trafic est rerouté vers Gemini ; l'incident reste ouvert jusqu'à acquittement.

UC-03 — Analyser les coûts et détecter le gaspillage de prompts

ActeurAdmin
PréconditionUn volume d'appels LLM a été collecté.
Scénario nominal1. L'admin ouvre la page /analytics. 2. Le frontend interroge GraphQL (cost_by_provider, repeated_prompts). 3. Le système renvoie les coûts par provider et la liste des prompts répétés > 5 fois. 4. Le système estime le gaspillage et affiche la recommandation « Enable cache for repeated prompts ».
PostconditionL'admin dispose d'une recommandation d'optimisation chiffrée.

1.7 — Règles de gestion

Invariants métier que le système doit garantir en permanence.

CodeRègle
BR-01Un provider ne peut jamais être supprimé physiquement : seul un marquage deleted_at / is_active=false est autorisé (piste d'audit).
BR-02Un service possède au plus un provider primaire. L'ajout d'un second primaire est rejeté.
BR-03Une règle de fallback ne peut être créée que si le provider primaire référencé existe déjà dans le service.
BR-04Le budget mensuel d'un service ou d'un provider doit être strictement positif.
BR-05Le cycle de vie d'un incident est strictement OPEN → ACKNOWLEDGED → RESOLVED ; un incident résolu ne peut pas être rouvert.
BR-06Les identifiants (secrets) d'un provider sont immuables après création : leur modification passe par la création d'un nouveau provider.
BR-07Le secret d'API n'est jamais exposé en clair dans une réponse API, un log ou un événement.
BR-08Toute configuration est la source de vérité dans PostgreSQL ; l'état Kong en est dérivé et reconstructible.
BR-09Une métrique perdue est tolérable (cohérence éventuelle) ; une configuration partiellement écrite ne l'est pas (transaction ACID).

1.8 — Contraintes et hypothèses

Contraintes techniques

  • Stack imposée : NestJS (monorepo), PostgreSQL, Redis, Kong 3.x, React + Vite.
  • Protocoles imposés : les cinq protocoles (REST, GraphQL, SSE, WebSocket, Webhook) doivent être présents et justifiés.
  • Conteneurisation : tout composant tourne sous Docker Compose.
  • Kong non modifié : Kong est utilisé tel quel ; aucune logique métier n'y est embarquée.

Contraintes projet

  • Équipe : 6 développeurs, un lot fonctionnel par personne.
  • Durée : itération courte (mode « video-coding », 1 à 2 jours par module sur le chemin critique).
  • Livrable final : MVP démontrable de bout en bout + documentation + démo.

Hypothèses

  • Les fournisseurs LLM exposent le nombre de tokens dans leurs réponses.
  • Le trafic de démonstration peut être simulé.
  • Un seul environnement (développement) est requis pour le MVP.

Partie II

Architecture détaillée

2.1 — Vision architecturale et style

Sentinel Gateway adopte un style hybride : plan de contrôle événementiel adossé à un noyau d'orchestration synchrone (Event-Driven Control Plane + Synchronous Orchestration Hub). Ce choix répond à une tension fondamentale du domaine : les métriques doivent circuler vite et peuvent tolérer la perte, tandis que les incidents exigent une cohérence immédiate.

Flux asynchrone
Métriques et événements transitent par Redis Pub/Sub. Perte tolérée, agrégation différée, débit privilégié.
Flux synchrone
Incidents et mises à jour Kong sont coordonnés en bloquant : pas de cohérence éventuelle, retour immédiat.
Découplage modulaire
Chaque module NestJS est autonome et communique via un EventBus, ce qui permet le travail parallèle des 6 développeurs.

Les cinq décisions d'architecture structurantes

#DécisionJustification
1Kong en data plane, Sentinel en control planeKong gère le routage HTTP éprouvé ; Sentinel se concentre sur l'intelligence. Économise 30-40 % d'effort d'ingénierie.
2Ségrégation des protocoles par cas d'usageChaque protocole est optimisé pour un motif d'accès précis : REST pour le CRUD, GraphQL pour l'analytique, SSE pour le flux, WebSocket pour la collaboration, Webhook pour l'événementiel sortant.
3Métriques événementielles, incidents synchronesLes métriques sont agrégées et peuvent être lossy ; les incidents requièrent un retour immédiat aux admins.
4PostgreSQL durable + Redis éphémèreLa configuration exige l'ACID ; les métriques admettent la cohérence éventuelle. Améliore le débit de 60-80 %.
5Salle d'incident = machine à états WebSocketÉtat central de l'incident, collaboration temps réel, mises à jour de fallback synchrones évitant les conditions de course.

2.3 — Architecture en couches

La responsabilité est stratifiée en cinq couches. Une couche ne dépend que de la couche immédiatement inférieure ou de l'EventBus transverse.

CoucheNomRôle
L1API Gateway (Kong)Routage HTTP, authentification de bord, rate limiting, transfert vers les upstreams.
L2Control Plane (Sentinel)Configuration, collecte de métriques, gestion des incidents, coordination des événements.
L3Temps réelSSE, WebSocket, Redis Pub/Sub : diffusion des métriques et coordination des incidents.
L4AnalytiqueAPI GraphQL : requêtes complexes pour les tableaux de bord.
L5PrésentationSPA React : rendu de l'interface, gestion de l'état client.

Règle de dépendance. Le code métier (couche L2) ne référence jamais Kong directement : il passe par la librairie kong-adapter. Cela empêche la logique spécifique à Kong de « fuir » dans les services métier (principe d'inversion de dépendance).

Sentinel est implémenté en monorepo NestJS géré par Nx. Le monorepo sépare les applications déployables (apps/) des librairies partagées et réutilisables (libs/). Ce découpage est aligné sur la répartition des 6 développeurs : chacun possède une ou plusieurs applications et les librairies associées.

2.14 — Sécurité et architecture frontend

Architecture de sécurité

  • Authentification : JWT avec expiration 8 h, vérifié par un guard NestJS sur REST, GraphQL et à la connexion WebSocket.
  • Secrets : chiffrement AES-256-GCM, clé maîtresse injectée par variable d'environnement (SENTINEL_MASTER_KEY).
  • Transport : HTTPS / TLS sur toutes les surfaces exposées.
  • Validation : validation stricte des entrées (class-validator / Joi) sur chaque DTO ; rejet 400 explicite.
  • Webhooks : charge signée HMAC pour authentifier l'émetteur côté récepteur.
  • Audit : toute mutation journalisée (auteur, nature, horodatage), rétention 2 ans.
  • Erreurs : jamais de stack trace exposée ; un correlation_id est renvoyé au client.

Architecture frontend (P6)

  • Stack : React 18 + Vite + TypeScript, Tailwind CSS, shadcn/ui, Recharts.
  • État : Apollo Client (état serveur GraphQL), état local léger pour l'UI.
  • Temps réel : client SSE (EventSource) pour les métriques, client Socket.io pour la salle d'incident.
  • Pages : /dashboard, /services, /providers, /analytics, /incident-room.
  • Performance : découpage de code, mémoïsation, skeleton loaders en cas de latence API.
  • Résilience UI : indicateur hors-ligne si SSE tombe ; salle d'incident désactivée si WebSocket indisponible.
  • Accessibilité : WCAG 2.1 AA — HTML sémantique, ARIA, navigation clavier.

Scalabilité horizontale. Kong (3→N nœuds), Sentinel (2→N instances synchronisées par Redis Pub/Sub), réplicas PostgreSQL en lecture, Redis en cluster. Chaque couche se met à l'échelle indépendamment, conformément à NFR-03.

3.1 — Principes d'organisation

Parallélisation
Après le gel des contrats (fin de phase 0), les 6 lots progressent indépendamment.
Contrats d'abord

Intégration continue
Daily standup, revues croisées, branche develop intégrée quotidiennement.

Stratégie d'intégration

  • Bouchons (mocks) : tant qu'une dépendance n'est pas livrée, chaque équipe travaille contre un bouchon respectant le contrat shared-types.
  • Branches : main protégée, develop intégrée, feature/* par lot avec PR obligatoire et revue d'un pair.
  • Portes d'architecture : revue d'architecture avant chaque jalon majeur pour vérifier le respect des contrats.

3.7 — Stratégie de tests

NiveauOutilsPérimètre & cible

IntégrationSupertest + base de testEndpoints REST, résolveurs GraphQL, transactions. Cible : couverture > 60 %.
Bout-en-boutCypress / PlaywrightParcours critiques : connexion → configuration → requête → dashboard, et flux d'incident complet.

3.8 — Definition of Done et jalons

Definition of Done (par tâche)

  • Code revu et approuvé par un pair (PR mergée dans develop).
  • Tests unitaires écrits et passants ; couverture du lot maintenue > 80 %.
  • Contrat shared-types respecté ; aucun couplage hors EventBus / contrat.
  • Validation des entrées et gestion d'erreurs implémentées.
  • Documentation à jour (OpenAPI / schéma GraphQL / README du module).
  • Le composant démarre sous Docker Compose sans intervention manuelle.

Jalons

  • J0 — Contrats gelés : shared-types, schéma DB et schémas d'API figés ; monorepo et CI en place.
  • J1 — Fondations vertes : Kong route le trafic, REST répond, ingestion fonctionne, layout React affiché.
  • J2 — Cœur intégré : CRUD complet, fallback opérationnel, salle d'incident fonctionnelle.
  • J3 — Temps réel de bout en bout : métriques live au dashboard, GraphQL analytique, événements distribués.
  • J4 — Gel fonctionnel : tous les lots terminés, périmètre MVP figé.
  • J5 — Démo validée : scénario complet rejoué, critères AC-01 à AC-10 satisfaits.

Erreurs de calcul de coûtMoy.MoyenTests unitaires des stratégies, validation manuelle avant la démo.

Désalignement de l'équipeFaibleMoyenDaily standup, portes d'architecture, revues croisées.
